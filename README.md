# Mingla — AI-Powered Experience Discovery Platform

Mingla is a full-stack, AI-driven experience discovery platform that recommends personalized activities, events, and places to users based on their preferences, location, social context, budget, and travel constraints. The platform ships as a **React Native (Expo) mobile app**, a **React admin/curator web dashboard**, and a **Supabase backend** powered by PostgreSQL, Edge Functions, and real-time subscriptions.

## Project Overview

Mingla combines **location-aware discovery**, **AI personalization**, and **real-time social collaboration** into a single mobile experience. Users swipe through curated experience cards generated by AI, plan outings with friends on collaborative boards, manage their calendar, message connections, and track their activity — all within a cohesive orange-themed UI.

### What Makes Mingla Different

| Capability | Details |
|---|---|
| **AI-Generated Cards** | Every recommendation card is produced by OpenAI GPT-4 / GPT-4o using real Google Maps Places data, Eventbrite events, weather forecasts, and the user's learned preference profile. |
| **Collaborative Sessions** | Users create named sessions, invite friends, swipe cards together in real time, vote/RSVP on a shared board, and chat within card-level discussion threads. |
| **Discover Tab** | A daily-refreshing, location-aware feed of curated experiences organized by categories, holidays, and night-out options — with city-specific AI enrichment. |
| **Behavioral Learning** | Every swipe, view, save, and share is recorded via `user_interactions`. A trigger-driven `user_preference_learning` table continuously refines the user's affinity scores across categories, times, and price bands. |
| **Multi-Account Types** | Supports **Explorer** (consumer) and **Curator** (content creator) account roles with distinct sign-up flows. |

### Row Level Security

Every table has RLS enabled with policies enforcing:
- **Ownership**: Users can only read/write their own data
- **Session membership**: Board/session data accessible only to participants
- **Friendship**: Profile visibility respects block lists
- **Authenticated role**: Insert/update restricted to `authenticated` role
- **SECURITY DEFINER functions**: Trigger functions that cross RLS boundaries (e.g., preference learning from interactions)

### Triggers

| Trigger | Effect |
|---|---|
| `update_updated_at_column` | Auto-updates `updated_at` on any row modification |
| `update_user_preferences_from_interaction` | AFTER INSERT on `user_interactions` → updates `user_preference_learning` with affinity scores (SECURITY DEFINER) |
| `accept_friend_request` | Auto-creates reciprocal friendship on acceptance |
| `add_board_card` | Side effects when cards are added to boards |
| `handle_board_vote` | Processes vote aggregation |
| `auto_create_profile` | Creates profile row on new user sign-up |
| `cleanup_session_under_two_participants` | AFTER DELETE on `session_participants` → auto-deletes sessions below 2 members |
| `sync_invite_inviter_ids` | Keeps `inviter_id` and `invited_by` columns in sync |

## Authentication & Security

### Auth Methods

| Method | Status |
|---|---|
| Email + Password | Active |
| Phone + OTP (SMS) | Active |
| Google OAuth | Active (WebView flow on mobile) |
| Apple Sign-In | Active (native on iOS) |

### Security Layers

1. **Supabase Auth (JWT)** — Every API call carries a JWT; `auth.uid()` is the single source of identity.
2. **Row Level Security (RLS)** — All 30+ tables have RLS enabled. Policies enforce ownership, session membership, and friendship-based visibility.
3. **SECURITY DEFINER Functions** — Trigger functions that need to cross RLS boundaries (e.g., writing to `user_preference_learning` from a `user_interactions` insert trigger) are declared SECURITY DEFINER with explicit `search_path`.
4. **Edge Function JWT Verification** — Most Edge Functions require a valid JWT (`verify_jwt = true`); only public proxies (weather, places, events) are open.
5. **OAuth Redirect Page** — Dedicated `oauth-redirect/` static page handles OAuth callback token exchange.
6. **Account Deletion** — `delete-user` Edge Function cascades through all user data across all tables.
7. **Blocked User Isolation** — Blocking a user cascades to hide profiles, prevent messages, and remove from conversation views via RLS policies.

## Getting Started

### Prerequisites

- **Node.js** >= 18
- **npm** (bundled with Node) or **Bun**
- **Expo CLI** — `npx expo`
- **EAS CLI** — `npm install -g eas-cli` (for builds)
- **Supabase CLI** — `npm install -g supabase` (for migrations & functions)

### 1. Clone the Repository

```bash
git clone https://github.com/your-org/Mingla.git
cd Mingla
```

### 2. Install Mobile App Dependencies

```bash
cd app-mobile
npm install
```

### 3. Environment Configuration

Create `app-mobile/.env` with your Supabase credentials:

```env
EXPO_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
EXPO_PUBLIC_SUPABASE_ANON_KEY=your-anon-key
```

### 4. Start the Development Server

```bash
cd app-mobile
npx expo start
```

Scan the QR code with Expo Go (iOS/Android) or press `a` for Android emulator / `i` for iOS simulator.

### 5. Build for Testing (APK / Simulator)

```bash
# Android APK (preview)
npx eas build --platform android --profile preview

# iOS simulator build
npx eas build --platform ios --profile development
```

## Development Workflows

### Mobile App

```bash
cd app-mobile
npx expo start                  # Development server
npx expo start --clear          # Clear cache and start
npx expo run:android            # Run on Android device/emulator
npx expo run:ios                # Run on iOS simulator
npx tsc --noEmit                # Type-check without building
npx expo lint                   # ESLint
```

### Database Migrations

```bash
cd Mingla   # root directory (where supabase/ folder is)

# Push all migrations to linked Supabase project
echo "y" | npx supabase db push --linked --include-all

# Create a new migration
npx supabase migration new my_migration_name

# Reset local database
npx supabase db reset
```

### Edge Functions

```bash
# Deploy a specific function
npx supabase functions deploy function-name

# Deploy all functions
npx supabase functions deploy

# Serve locally for development
npx supabase functions serve function-name --env-file .env.local
```

### Web Dashboard

```bash
cd "Mingla App (Admins, QA, CM, and Curators)"
npm install
npm run dev          # Vite dev server
npm run build        # Production build
```

## Deployment

### Mobile App (EAS Build)

```bash
# Production builds
npx eas build --platform android --profile production
npx eas build --platform ios --profile production

# Submit to stores
npx eas submit --platform android
npx eas submit --platform ios
```

### Backend (Supabase)

- **Database**: Managed by Supabase. Migrations applied via `supabase db push`.
- **Edge Functions**: Deployed via `supabase functions deploy`.
- **Storage**: Avatars bucket configured with RLS policies.
- **Auth**: Configured in Supabase Dashboard (Google, Apple providers, email/phone settings).

### Web Dashboard

Deploy to **Vercel**, **Netlify**, or any static hosting:

```bash
cd "Mingla App (Admins, QA, CM, and Curators)"
npm run build
# Deploy dist/ folder
```
