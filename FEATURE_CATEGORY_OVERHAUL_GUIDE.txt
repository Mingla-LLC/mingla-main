================================================================================
COMMON MISTAKES TO AVOID
================================================================================

1. DON'T mix slugs and display names in the PreferencesSheet categories array.
   The `id` field in that array is what gets stored in the DB. Use display-name
   IDs like 'Fine Dining', not 'fine_dining'. The edge function handles both
   via its multi-variation CATEGORY_MAPPINGS.

2. DON'T forget to update the EXCLUDED_TYPES as well as CATEGORY_MAPPINGS in
   the edge function. Missing exclusions will cause irrelevant venues to appear.

3. DON'T run the DB migration before the code is live. Users' preferences will
   be migrated to new names, but if the old code is still running it will get
   confused by the new names in the DB.

4. DON'T assume chef_led_restaurant, upscale_restaurant, cinema, cold_plunge_facility,
   or float_tank_center are real Google Places API types. They are not. The API
   silently ignores them — this is fine, but don't add them expecting results.

5. DON'T forget to update OnboardingFlow.tsx. New users going through onboarding
   will write their selected categories directly to the DB. If OnboardingFlow
   still uses old category names, new users will have migrated-already values
   written over with old names.

6. DO deploy the edge function changes BEFORE running the migration. Edge
   functions need to understand new category names before user preferences
   contain them.
