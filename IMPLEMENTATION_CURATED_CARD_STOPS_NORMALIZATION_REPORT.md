# Implementation Report: Curated Card Stops Normalization

## A. Target Schema

```
place_pool (source of truth)
  └── card_pool_stops (FK ON DELETE CASCADE to place_pool.id)
        └── card_pool (FK ON DELETE CASCADE from card_pool_stops.card_pool_id)
```

**card_pool_stops:**
| Column | Type | Constraints |
|--------|------|-------------|
| id | UUID PK | |
| card_pool_id | UUID NOT NULL | FK → card_pool(id) ON DELETE CASCADE |
| place_pool_id | UUID NOT NULL | FK → place_pool(id) ON DELETE CASCADE |
| google_place_id | TEXT NOT NULL | Denormalized for query convenience |
| stop_order | INTEGER NOT NULL | CHECK >= 0 |
| | | UNIQUE(card_pool_id, stop_order) |
| | | UNIQUE(card_pool_id, place_pool_id) |

## B. Delete Semantics

```
place_pool row deleted
  → card_pool_stops rows with that place_pool_id CASCADE deleted
  → trigger (trg_delete_curated_card_on_stop_loss) fires per deleted stop
  → trigger deletes the parent card_pool row (WHERE card_type='curated')
  → card_pool CASCADE deletes remaining card_pool_stops rows + user_card_impressions
```

**No partial survival.** Any stop loss = full curated card deletion.

## E. Edge Cases

- **Trigger recursion:** When the trigger deletes a curated card, that CASCADE deletes remaining stops, which fires the trigger again for those stops. The second trigger call is a no-op because the parent card is already deleted.
- **Concurrent deletes:** Two place_pool rows deleted simultaneously that both belong to the same curated card — first trigger deletes the card, second trigger's DELETE is a no-op (0 rows affected). Safe.
- **Null/empty stops during backfill:** Curated cards with NULL or empty `stop_place_pool_ids` are deleted during migration (Step 3b).

## F. Rollback Considerations

- **Before running:** The migration deletes curated cards with orphaned stops. This is intentional and irreversible — those cards were already broken.
- **To roll back the schema change:** You would need to re-add the dropped columns and repopulate from `card_pool_stops`. A reverse migration script would be needed.
- **Curated cards are regeneratable.** If the migration deletes cards, they can be regenerated by running the curated experience generation pipeline.

## G. Regeneration After Deletion

When curated cards are deleted (either by this migration or by future place_pool deletions), the `generate-curated-experiences` edge function will regenerate them on the next request for that area/experience type. The pool-first flow (`serveCuratedCardsFromPool`) will find fewer cards → fall through to generation → new cards with valid stops get inserted into both `card_pool` and `card_pool_stops`.

No manual intervention needed. The pipeline is self-healing.
