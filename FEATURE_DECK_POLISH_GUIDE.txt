STEP 1: FIX isFullBatchLoaded DEADLOCK (Critical — No Cards Bug)
================================================================================

FILE: app-mobile/src/hooks/useDeckCards.ts
LINE: 78

BEFORE:
  isFullBatchLoaded: (query.data?.total ?? 0) >= (20 * 0.5),

AFTER:
  isFullBatchLoaded: !query.isLoading && !query.isFetching && query.data !== undefined,

WHY: The old threshold required >= 10 cards. When a location yields 0-9
cards, or when edge functions fail, the batch was NEVER marked loaded.
This caused:
  - RecommendationsContext sync effect (line 419-437) never cleared
    isBatchTransitioning
  - Completion effect (line 551-591) never marked fetch complete
  - UI showed permanent loading spinner

The new check simply means "the query has settled" — whether it returned
20 cards or 0 cards. The existing empty-state handler at
RecommendationsContext.tsx:428-434 correctly handles the 0-card case.

TEST: Open app in a remote area with few Google Places results. Cards
should appear (even if fewer than 20) or show "No matches" state.

STEP 6: PERFORMANCE — CACHE LAST-KNOWN LOCATION
================================================================================

FILE: app-mobile/src/hooks/useUserLocation.ts

Add location caching to eliminate the 100-500ms wait for GPS/DB:

--- At the top of the hook: ---

  import AsyncStorage from '@react-native-async-storage/async-storage';

  const LOCATION_CACHE_KEY = '@mingla/lastLocation';

--- In the query config, add initialData: ---

  // Outside the hook, create a sync reader:
  let cachedLocationSync: { lat: number; lng: number } | null = null;
  AsyncStorage.getItem(LOCATION_CACHE_KEY).then(raw => {
    if (raw) {
      try {
        const parsed = JSON.parse(raw);
        if (parsed.lat && parsed.lng) cachedLocationSync = parsed;
      } catch {}
    }
  });

  // Inside useQuery:
  initialData: cachedLocationSync
    ? { lat: cachedLocationSync.lat, lng: cachedLocationSync.lng }
    : undefined,

--- After successful location resolution: ---

  // In the onSuccess or in an effect watching the data:
  useEffect(() => {
    if (data?.lat && data?.lng) {
      AsyncStorage.setItem(
        LOCATION_CACHE_KEY,
        JSON.stringify({ lat: data.lat, lng: data.lng, ts: Date.now() })
      ).catch(() => {});
    }
  }, [data?.lat, data?.lng]);

WHY: On second+ app launches, the deck query fires instantly with cached
location. Fresh GPS updates in background. If the user hasn't moved
significantly, the same cards serve (cache hit). If they moved, deck
invalidates when fresh location differs by > ~500m.

TEST: Open app → see cards instantly. Kill app → reopen → cards appear
in < 1 second (no location wait).

COMMON MISTAKES TO AVOID
================================================================================

1. DO NOT change the round-robin algorithm — it's correct.
   The bug is in pill RESOLUTION (not getting pills for some categories)
   and BATCH LOADED detection (threshold too high).

2. DO NOT add new edge functions for Groceries & Flowers.
   Reuse `discover-experiences` with category filter.

3. DO NOT change how categories are saved to DB.
   The mixed intents+categories array format works correctly.
   separateIntentsAndCategories() splits them properly.

4. DO NOT modify the card pool tables or pipeline.
   The pool infrastructure is correct. The fix is on the mobile side
   (useDeckCards threshold, pill resolution).

5. DO NOT change the RecommendationsContext sync effect logic
   EXCEPT the isDeckBatchLoaded dependency (which is fixed by Step 1).
