2: The spec is written. Here's the executive summary:

What Was Found
Three independent failure modes — any one of them alone causes the infinite loader:

Failure Mode 1 — Native API hang (enhancedLocationService.ts)
getLastKnownLocation() calls Location.getLastKnownPositionAsync() with no timeout. On Android and some iOS simulators this call blocks indefinitely. GPS has a 10s timeout; the fallback after GPS does not. Result: isLoadingLocation stuck true forever.

Failure Mode 2 — Async chain hang (useUserPreferences.ts)
When Supabase fails (30s timeout or network cut), it falls through to offlineService.getOfflineUserPreferences() — an AsyncStorage read with no timeout. If AsyncStorage is slow or corrupted, isLoadingPreferences stays true forever.

Failure Mode 3 — Logic dead zone (RecommendationsContext.tsx)
Even when both queries resolve cleanly with null, the loader keeps spinning. The "Mark Fetch Complete" effect has three branches — all three require either userLocation ≠ null, or recommendations.length > 0, or locationError truthy. A null location (returned without error) satisfies none of them. And the 5-second safety timeout only fires on mode changes, not initial load (previousModeRef.current === undefined on first render → modeChanged = false).

The Three-Layer Fix
Layer	File	What It Does
1 — Root cause	enhancedLocationService.ts	3s timeout on getLastKnownLocation()
1 — Root cause	useUserPreferences.ts	8s total timeout on the full fetch chain
1 — Root cause	useUserLocation.ts	13s total timeout (10s GPS + 3s buffer)
2 — Logic fix	RecommendationsContext.tsx	allPrerequisitesSettled branch: fires the moment all three isLoading flags drop to false, regardless of whether they returned data or null
2 — Logic fix	RecommendationsContext.tsx	effectivePrefs fallback: when prefs fail, deck starts with ["Nature", "Casual Eats", "Drink"] instead of staying permanently gated
3 — Nuclear	RecommendationsContext.tsx	15s mount-only safety timeout using a useRef (avoids stale closure) — the catch-all for any future regression

3: FEATURE_BOARD_COLLABORATORS_SPEC.md is done. Here's what the deep analysis found that wasn't in the original spec:

Three additional findings incorporated:

session_id is nullable for a real reason — solo boards exist, and boardService.createBoard() can create boards with no session. The fallback RLS path handles this without ever blocking valid use. The fast path fires for 99%+ of real traffic (session-originated boards).

The INSERT RLS cannot use auth.uid() = user_id — the accepting user inserts rows for ALL participants including the session creator. The policy must check session membership, not identity. This was the subtle trap — getting it wrong would mean the owner row silently fails to insert.

The race condition is real but partially guarded — concurrent accepts can create orphaned boards rows. The partial unique index on collaboration_sessions(board_id) blocks two sessions from pointing at the same board and prevents the worst-case data corruption, but orphaned boards are a pre-existing application-layer problem. Documented in §10.2 with a future-proof note.

Three code changes are required (not zero as the previous spec stated) — the session_id column must be populated by the caller at insert time. All three changes are additive, backward-compatible, and surgical.

4: The spec is at FEATURE_ACCOUNT_STABILITY_AND_AUTH_FIX_SPEC.md.

Here's what the investigation found — three separate log signatures, two real bugs, one non-issue:

Non-Issue: warmPool / discover-cards 15s warnings
These are working exactly as designed. Both warmPool and the discover-cards warm call use Promise.race with a deliberate 15-second timeout as a safety valve against hung Supabase edge function cold starts. The warnings are swallowed with .catch(() => {}). Zero user impact.

Bug A — CRITICAL: useAuthSimple is being called from 11 components simultaneously

This is a fundamental architectural problem. useAuthSimple is a plain hook, not a context or singleton. Every component that calls it independently runs supabase.auth.getSession() (a network call) and opens its own supabase.auth.onAuthStateChange subscription. On a typical screen, 6–8 of these are mounted at the same time. That's 6–8 parallel session fetches on every mount cycle, and 6–8 live auth subscriptions firing in lockstep every time auth state changes. The duplicate [AUTH] Initializing — fetching session... in your log is direct evidence of this — it fires once per mounted component, not once per app session.

The fix is mechanical: 10 components need to switch from const { user } = useAuthSimple() to const { user } = useAppStore(). The user is already written into Zustand by AppStateManager. Everyone else should read from the store.

Bug B — HIGH: delete-user N+1 query pattern causes the 45-second TIMEOUT

cleanupCollaborationSessions() loops over each session the user is admin/creator of, firing individual DB queries sequentially per session. For a user in 5 sessions as admin: ~50 sequential round-trips at ~50ms each = 2.5 seconds just for that function, before 30+ table deletes, the profile RPC call, and the auth admin delete. The 45-second wall-clock timeout fires because the edge function simply cannot complete in time.

The fix rewrites the function to use 7–9 queries total regardless of session count — fetch all sessions once, fetch all participants once, build promotion/transfer writes in memory, execute them in a single Promise.allSettled, then batch-delete under-populated sessions using IN clauses instead of per-session loops.

Bug C — MEDIUM: Silent sign-out on orphaned delete promise

If a user force-closes the delete modal (10-second escape hatch) while the edge function is still in-flight, and the edge function later succeeds, the orphaned promise calls handleSignOut() — signing the user out without warning. The fix is one abortedRef flag checked immediately after Promise.race resolves.



5: he spec is written at FEATURE_ACCOUNT_DELETION_FIX_SPEC.md.

Here's what the investigation found — three independent defects that compound each other:

Root cause chain:

Wrong deletion order in delete-user — supabase/functions/delete-user/index.ts:469-515: Profile (public.profiles) is deleted in Step 3, but auth.users isn't deleted until Step 4. This creates a 1-2 second window where the client has a valid JWT but no profile row. Every PGRST116 error you see flows from this window.

No global signout lock in useAuthSimple — app-mobile/src/hooks/useAuthSimple.ts:93-111 and lines 187-199: The hook is mounted in 4+ components simultaneously (profile tab, AccountSettings modal, etc.). Each instance registers its own onAuthStateChange listener AND calls initializeAuth(). When PGRST116 hits, all 4 instances call signOut() concurrently — producing 6+ SIGNED_OUT events and 6+ WelcomeScreen renders.

Location tracking has no auth listener — app-mobile/src/services/enhancedLocationTrackingService.ts:208-236: The service keeps running after profile deletion. It calls supabase.auth.getUser(), gets the user back (auth.users still valid), then tries to insert into user_location_history — which FK-references the already-deleted profile row → constraint violation.

The three fixes:

Flip Steps 3 & 4 in the edge function (delete auth first → immediately kills the JWT → stops all client operations)
Add a module-level _isSigningOutDueToMissingProfile boolean to useAuthSimple so only one instance handles the teardown
Wire supabase.auth.onAuthStateChange in the location service to call stopLocationTracking() on SIGNED_OUT
