================================================================================
 CARD REFRESH LOADER SYNC & SPEED OPTIMIZATION — IMPLEMENTATION GUIDE
================================================================================

PROBLEM: When user saves preferences, the overlay spinner stops 1-3 seconds
before all cards actually load. The spinner tracks only the recommendations
query, ignoring curated and nature queries. Also, redundant cache invalidations
trigger double-fetches, wasting network and time.

GOAL: Spinner stays visible until ALL data pipelines settle. ≤3 second load.

3 FILES TO CHANGE. Zero new files. Zero DB changes. Zero edge function changes.

 STEP 3: AppHandlers.tsx — Remove Redundant Invalidations
================================================================================

File: app-mobile/src/components/AppHandlers.tsx

3a) REMOVE redundant recommendations invalidation (line 715):

    DELETE this line:
    ```
    queryClient.invalidateQueries({ queryKey: ["recommendations"] });
    ```

    WHY: The recommendations query key includes all preference values (budget,
    travel mode, etc). When prefs change, the query key changes automatically →
    React Query creates a new query. Invalidating the OLD key is wasted work.

3b) REMOVE redundant curated-experiences invalidation (line 716):

    DELETE this line:
    ```
    queryClient.invalidateQueries({ queryKey: ["curated-experiences"] });
    ```

    WHY: The refreshKey increment (line 747) triggers the effect in
    RecommendationsContext (line 528) that invalidates curated queries. This is
    a duplicate call.

3c) REMOVE redundant nature removeQueries (line 742):

    DELETE this line:
    ```
    queryClient.removeQueries({ queryKey: ['nature-cards'] });
    ```

    WHY: The nature query key includes all preference values. When prefs change,
    the key changes → new query. The hash-based resetNatureHistory() call
    (lines 739-741) already handles Zustand batch state. The removeQueries is
    redundant and can cause unnecessary cache thrashing.

    KEEP these lines (739-741):
    ```
    const { naturePrefsHash, resetNatureHistory } = useAppStore.getState();
    if (newHashStr !== naturePrefsHash) {
      resetNatureHistory(newHashStr);
    }
    ```

3d) KEEP the userLocation invalidation (line 717):
    ```
    queryClient.invalidateQueries({ queryKey: ["userLocation"] });
    ```
    This is needed because location re-evaluation may depend on GPS toggle change.

3e) KEEP the refreshKey increment (lines 746-750) — this is the master trigger.

RESULT after Step 3: On preference save, the cascade is:
    1. Optimistic cache update (instant)
    2. userLocation invalidation (triggers location re-fetch if GPS changed)
    3. refreshKey increment (triggers query key changes + curated invalidation)
    That's it. No redundant work.

HOW TO TEST STEP 3:

      - generate-curated-experiences: should be called ONCE per enabled type
        (not twice)
      - discover-nature: should be called ONCE (if nature selected)
      - new-generate-experience: should be called ONCE
    - Previously: you'd see duplicate calls for curated and recommendations

================================================================================
 COMMON MISTAKES TO AVOID
================================================================================

1. DO NOT put the clearing effect before `allBatchesLoaded` is defined.
   It must be placed after line 660 where allBatchesLoaded is computed.

2. DO NOT remove the curated-experiences invalidation in RecommendationsContext
   (line 528). Only remove the one in AppHandlers (line 716). The refreshKey-
   driven one in RecommendationsContext is the canonical trigger.

3. DO NOT change the safety timeout to less than 5 seconds. Edge functions
   with cold starts can take 3-4 seconds. 8 seconds gives enough headroom.

4. DO NOT add isRefreshingAfterPrefChange to the `isAnyLoading` flag on
   SwipeableCards line 209. That flag controls the FULL-SCREEN loader, which
   we don't want during preference refreshes (old cards should stay visible
   under the overlay). The overlay is the correct UX.

5. DO NOT remove `placeholderData` from the hooks. It's what keeps old cards
   visible during refresh. The fix works WITH placeholder data, not against it.
